## 1. Executive Summary

**Feasibility Verdict**: Feasible with Changes

**Complexity**: Medium

**Key Recommendations**:
- Use standard Vercel deployment (not `output: export`) to support dynamic routes (`/skills/[id]`) via `generateStaticParams`. All pages are pre-rendered at build time and served from Vercel's CDN — free tier compatible, no cold starts.
- Establish a strict RSC/Client Component boundary: all Framer Motion animated components must live in `components/client/` and begin with `'use client'`. All data loading (markdown parsing, config reading) stays in RSCs or build-time functions.
- Use a Node.js prebuild script (`scripts/copy-skills.mjs`) to copy `.agents/skills/` markdown files into `website/content/skills/` before `next build`. Wire it to the `"prebuild"` npm hook so Vercel runs it automatically.
- Use `next-mdx-remote/rsc` for markdown rendering inside App Router RSCs — it is the correct library variant for the RSC constraint and avoids client-side hydration overhead.
- Implement i18n with a lightweight custom React Context and two static locale JSON files rather than `next-intl`, avoiding middleware/adapter configuration overhead for a two-locale, small-string-count site.

**Major Risks**:
- Framer Motion component boundary leakage: importing an animated component into an RSC without an explicit `'use client'` boundary causes a build error or silent CSR fallback. Requires folder-level convention enforcement.
- Prebuild copy step failure on Vercel: if `next build` is invoked directly instead of `npm run build`, the `prebuild` hook does not fire and `website/content/skills/` will be empty, causing all `/skills/[id]` pages to 404 or the build to fail. Must verify the Vercel build command is set to `npm run build`.
- Skill config and markdown directory sync: if `website/data/skills.ts` and `.agents/skills/` directory entries diverge, `generateStaticParams` may reference IDs with no markdown file or omit valid skills. Requires a build-time validation check.

## 3. Technical Recommendations

### 3.1 Architecture Pattern

**Recommended Pattern**: Static Site Generation (SSG) with Hybrid RSC / Client Component Tree

**Justification**:
- The site has no user-specific data, no authentication, and no runtime mutations. SSG is the correct default for all pages.
- Next.js App Router makes RSC the default, which is correct for layout and build-time data loading. Client Components are opted-in only for interactivity or browser APIs (animations, clipboard, language switcher, URL filter state).
- All pages are pre-rendered HTML served from Vercel's CDN edge with zero server compute per request, keeping the free tier comfortably within limits.
- No API routes or server actions should be introduced in v1.

**Architecture Diagram**:

```
Monorepo root/
├── .agents/skills/            <- Source of truth for skill markdown
│         |
│         └── [prebuild: scripts/copy-skills.mjs]
│                   |
│                   v
└── website/                   <- Next.js 14 App Router app
    ├── content/skills/        <- Copied markdown files (gitignored)
    ├── data/                  <- TypeScript config: skills[], workflows[], tools[]
    ├── app/
    │   ├── page.tsx           <- RSC: Home
    │   ├── install/page.tsx   <- RSC shell + 'use client' tab/copy components
    │   ├── workflow/page.tsx  <- RSC shell + 'use client' DAG visualizer
    │   ├── skills/page.tsx    <- RSC shell + 'use client' filter + card grid
    │   └── skills/[id]/
    │       └── page.tsx       <- RSC: generateStaticParams + MDXRemote render
    ├── components/
    │   ├── server/            <- RSC-safe layout and structural components
    │   └── client/            <- 'use client' animated and interactive components
    └── lib/
        ├── skills.ts          <- Build-time reader for content/skills/
        └── i18n/              <- LocaleContext.tsx + en.json + vi.json
```

### 3.2 Technology Stack

| Layer | Technology | Reason |
|-------|------------|--------|
| Framework | Next.js 14 (App Router) | Specified in BA. SSG via `generateStaticParams`, RSC-first architecture |
| Language | TypeScript | Consistent with monorepo CLI codebase. Skills config as typed objects enables `generateStaticParams` type safety |
| Styling | Tailwind CSS + shadcn/ui | Specified in BA. shadcn/ui components are RSC-compatible by default; copied into the project so they do not change without an explicit update |
| Animation | Framer Motion | Specified in BA. Confined to Client Components; supports `prefers-reduced-motion` via `useReducedMotion` hook |
| Markdown rendering | next-mdx-remote/rsc | Correct variant for App Router RSC. Renders MDX server-side at build time without client hydration overhead |
| i18n | Custom React Context + JSON | Lightweight. Avoids next-intl middleware and adapter configuration for a two-locale, small-string-count site |
| Static data | TypeScript config files in `website/data/` | Single source of truth for skill IDs, workflow DAG definitions, and tool enum |
| Prebuild | Node.js ESM script (`scripts/copy-skills.mjs`) | Copies `.agents/skills/*/SKILL.md` into `website/content/skills/{id}.md` before `next build` |
| Hosting | Vercel (standard deployment, free tier) | Free tier supports full SSG. No `output: export` needed |
| Package manager | npm (existing monorepo convention) | Consistent with existing CLI project |

### 3.3 Reuse Opportunities

| Asset | Path | How to Use |
|-------|------|------------|
| Skill markdown source files | `.agents/skills/*/SKILL.md` | Copied at prebuild time into `website/content/skills/`; parsed with `next-mdx-remote/rsc` for `/skills/[id]` pages |
| Skill directory entries | `.agents/skills/` (folder names) | Used by the prebuild script to enumerate valid skill IDs; drives `generateStaticParams` and the TypeScript `skills.ts` config |
| Monorepo `package.json` | Root or `website/package.json` | Add `"prebuild": "node scripts/copy-skills.mjs"` to ensure the copy runs before `next build` on Vercel |

## 4. Technical Edge Cases

| ID | Category | Edge Case | Expected Behavior | Priority |
|----|----------|-----------|-------------------|----------|
| TE-01 | Data | A skill folder in `.agents/skills/` has no `SKILL.md` | Prebuild script skips the folder and logs a warning; `generateStaticParams` excludes that ID; no runtime 404 | Must |
| TE-02 | Data | Skill markdown contains raw HTML or unrecognized MDX JSX components | Use a restricted component map in `MDXRemote`; allow only safe HTML elements; unknown JSX tags render as null or a fallback | Must |
| TE-03 | Data | `website/data/skills.ts` and `.agents/skills/` directory are out of sync | A build-time validation script diffs config IDs against copied markdown files and prints warnings on mismatch, preventing silent dead links | Should |
| TE-04 | Data | Skill markdown frontmatter is missing or malformed | `lib/skills.ts` parses frontmatter defensively and applies field defaults rather than throwing at build time | Should |
| TE-05 | State | User switches language while the Skills Explorer has an active filter in the URL | Language switch updates Context; the URL query param filter is preserved and not reset | Should |
| TE-06 | State | User bookmarks `/skills?tool=codex` and returns later | `useSearchParams` reads the query param on mount and applies the filter immediately; no flash of unfiltered content | Must |
| TE-07 | Security | URL query param for the skill filter receives an unrecognized tool value | Filter value is matched against the closed `ToolId` enum from the TypeScript config; unrecognized values are silently ignored and the filter resets to "All" | Must |
| TE-08 | Compatibility | `localStorage` is unavailable (SSR context, private browsing with storage blocked) | Language preference read from `localStorage` is wrapped in a try/catch and falls back to the default locale (English); no SSR crash | Must |
| TE-09 | Performance | `/skills` page renders more than 50 skill cards with filter interaction | Filter is a synchronous array operation over in-memory data — no loading state needed. If card count exceeds 100, apply `React.memo` to skill cards to avoid full re-renders on filter change | Should |
| TE-10 | Performance | DAG visualizer on `/workflow` with many nodes causing animation jank | Keep node count under 20 in v1. For larger DAGs, apply scroll/zoom via a single CSS transform on the container rather than individual Framer Motion nodes | Should |
| TE-11 | Build | `website/content/skills/` does not exist when `next build` runs on a fresh Vercel deployment | The `"prebuild"` npm hook fires before `"build"` when Vercel uses `npm run build` as the build command. The copy script must also create the directory if it does not exist. Verify in Vercel build logs on first deploy. | Must |
| TE-12 | Animation | User has `prefers-reduced-motion: reduce` set in OS accessibility settings | All Framer Motion animated components must check `useReducedMotion()` and skip or reduce animation variants accordingly | Must |

## 5. Risk Assessment

### Technical Risks

| ID | Risk | Impact | Likelihood | Mitigation |
|----|------|--------|------------|------------|
| TR-01 | Framer Motion component imported into an RSC without a `'use client'` boundary | High — build error or broken animation in production | Medium — easy mistake in a shared component file | Enforce `components/client/` folder convention where every file begins with `'use client'`. Consider an ESLint custom rule or CI check. |
| TR-02 | Prebuild copy step does not run on Vercel (build command set to `next build` directly) | High — all `/skills/[id]` pages return 404 or build fails | Low if `"prebuild"` npm hook is configured correctly and the Vercel build command is `npm run build` | Explicitly set Vercel build command to `npm run build`. Verify the `prebuild` hook appears in Vercel build logs on first deployment. |
| TR-03 | `next-mdx-remote/rsc` API changes between minor versions | Medium — build breaks, markdown does not render | Low — library is stable and widely used | Pin `next-mdx-remote` to a specific minor version in `package.json`. Review changelog before upgrading. |
| TR-04 | DAG layout becomes unmanageable with manual coordinate config as the workflow grows | Medium — visualizer accuracy degrades | Medium — if the workflow definition grows beyond 15 nodes | Design the workflow config schema to accept `x`/`y` or grid `row`/`col` coordinates from the start. Document the schema clearly so adding nodes is self-service. |
| TR-05 | Skill config and markdown directory diverge over time | Medium — dead links or missing pages | Medium — as new skills are added to `.agents/skills/` without updating `website/data/skills.ts` | Build-time validation script (TE-03). Document the two-step process: add folder to `.agents/skills/` AND add entry to `website/data/skills.ts`. |
| TR-06 | i18n locale JSON grows large and contributes to initial bundle | Low — small site with a limited string set | Low | Keep locale JSON split per route/section if bundle size becomes a concern. Defer this optimization to Phase 3. |

### Dependency Risks

| ID | Dependency | Risk | Mitigation |
|----|------------|------|------------|
| DR-01 | `next-mdx-remote` | Library drops RSC support or changes the `rsc` import path in a future version | Pin version. Abstract rendering behind `lib/mdx.ts` so the library is swapped in a single file. |
| DR-02 | Framer Motion | Major version break changes animation API | Pin major version. Framer Motion maintains a stable public API with long deprecation windows. |
| DR-03 | shadcn/ui components | Component designs or APIs change in a future shadcn/ui version | Not a runtime dependency — components are copied into the project on install and are owned by the codebase. They will not change unless explicitly updated. |
| DR-04 | Vercel free tier | Bandwidth or build minute limits change | Site is lightweight static HTML with minimal bandwidth. Free tier is appropriate indefinitely for this use case. |

## 6. Implementation Guidance

### 6.1 Suggested Phases

#### Phase 1: Foundation
**Priority**: High | **Complexity**: Low

| Task | Description |
|------|-------------|
| Website project setup | Create `website/` folder in monorepo root. Initialize Next.js 14 App Router with TypeScript, Tailwind CSS, shadcn/ui CLI install. |
| Prebuild copy script | Write `scripts/copy-skills.mjs`. Reads `.agents/skills/*/SKILL.md`. Copies to `website/content/skills/{id}.md`. Creates directory if absent. Wire to `"prebuild"` npm hook in `website/package.json`. |
| TypeScript data config | Create `website/data/skills.ts` (SkillConfig array), `website/data/workflows.ts` (WorkflowConfig with nodes/edges), `website/data/tools.ts` (ToolId enum + labels). |
| i18n foundation | Create `website/lib/i18n/LocaleContext.tsx`, `en.json`, `vi.json`. Wrap root `layout.tsx` with the provider. |
| Component boundary convention | Create `website/components/server/` and `website/components/client/` folders. Add a brief convention README. |
| Vercel project setup | Connect `website/` subfolder to Vercel. Set build command to `npm run build`. Set root directory to `website/`. Confirm `prebuild` hook fires in first build log. |

#### Phase 2: Core Features
**Priority**: High | **Complexity**: Medium

| Task | Description |
|------|-------------|
| Home page | RSC layout shell + Client Component hero section with Framer Motion entrance animation + static feature highlight cards |
| `/install` page | RSC shell with static install instructions. Client Component for AI tool tab switcher and copy-to-clipboard button using the browser Clipboard API. |
| `/skills` explorer | RSC shell. Client Component for single-select tool filter synced to URL query param via `useSearchParams`. Skill card grid rendered from `website/data/skills.ts`. |
| `/skills/[id]` detail | RSC page with `generateStaticParams` reading IDs from `website/data/skills.ts`. Markdown body rendered via `next-mdx-remote/rsc` reading from `website/content/skills/`. |
| `/workflow` visualizer | RSC shell passing workflow config as props. Client Component DAG visualizer with Framer Motion step-by-step animation. Node coordinates from `website/data/workflows.ts`. |

#### Phase 3: Enhancement
**Priority**: Medium | **Complexity**: Low

| Task | Description |
|------|-------------|
| Language switcher | Client Component in nav. Reads/writes `localStorage` with try/catch guard. Integrates with `LocaleContext`. |
| Build-time validation | Node.js script that diffs skill IDs in `website/data/skills.ts` against files in `website/content/skills/` and prints mismatches. Run as part of the `prebuild` step. |
| SEO metadata | Add `generateMetadata` per page for `title`, `description`, and `og:image`. |
| Reduced motion support | Add `useReducedMotion()` check to all Framer Motion Client Components. Provide static fallback variants. |
| Performance audit | Run Lighthouse on deployed site. Apply `React.memo` to skill cards if filter re-renders are slow. Check for layout shift caused by animations. |
| Accessibility pass | Ensure `aria-label` on icon-only buttons, keyboard navigation on tabs, focus management after language switch. |

### 6.2 Technical Constraints

| Constraint | Description | Impact |
|------------|-------------|--------|
| No runtime API calls | All data resolved at build time | No `fetch()` in component bodies. Data comes from imported TypeScript modules or `fs.readFile` in RSCs (which runs at build time during SSG). |
| Vercel free tier | No server-side runtime compute per request | Confirms SSG-only approach. No API routes, no server actions. |
| Markdown source outside `website/` | `.agents/skills/` is not inside the Next.js project root | Requires prebuild copy step. Cannot reliably use `fs` paths outside the Next.js root in Vercel production builds. |
| Client Component opt-in | All Framer Motion usage requires `'use client'` | Props passed from RSC to Client Component must be fully serializable (no functions, no class instances, no `Date` objects unless stringified). |
| No global state library | Site scope does not justify Zustand or Redux | Use React Context only for i18n. Filter state lives in URL. Tab and copy state is local `useState`. |

## 8. Alternatives Considered

| Alternative | Pros | Cons | Why Not Chosen |
|-------------|------|------|----------------|
| `output: export` (true static export) | Deployable to any static CDN; no Vercel runtime at all | `useSearchParams` requires explicit `Suspense` wrapping in static export mode, adding boilerplate to every filtered page | Standard Vercel deployment handles SSG identically for all pages without the `Suspense` requirement. |
| `next-intl` for i18n | Full-featured: routing-based locale detection, plurals, date/number formatting | Requires Next.js middleware for route-based locale detection; overkill for two languages and a small string count | Custom Context and JSON is sufficient for the site's scope. |
| Gatsby instead of Next.js 14 | Excellent SSG, mature plugin ecosystem for markdown | BA specified Next.js; Framer Motion and shadcn/ui have first-class Next.js documentation | BA decision is firm: Next.js 14 App Router. |
| Reading `.agents/skills/` via `fs` directly in `generateStaticParams` (without prebuild copy) | Eliminates the copy step | Vercel builds sandbox the Next.js project root; paths outside the root are not reliably accessible in production | Prebuild copy is explicit, portable, and works identically in local dev and on Vercel. |
| `gray-matter` + `remark` pipeline (manual) | Full control over the markdown AST | More boilerplate; `next-mdx-remote/rsc` provides the same RSC-compatible pipeline with significantly less code | `next-mdx-remote/rsc` is the community-standard solution. |
